# LLMs.txt Design Guidelines

## Purpose

These files provide LLM-readable documentation indexes to help AI assistants quickly understand and reference this documentation site.

## File Specifications

### llms.txt (Concise Version)
**Purpose**: Quick reference and context loading for LLMs
**Max Size**: ~50KB recommended
**Content**:
- Site metadata (title, URL, description)
-- Title: AI Tutorial \[dot\] dev
-- URL: https://aitutorial.dev
-- Description: An Interactive AI Tutorial With Concepts and Leassons Learned. Learn by experimenting with real code in an interactive sandbox. No setup, no dependencies—just open your browser and start building. Every concept includes runnable examples across multiple frameworks, so you can try things immediately and see what actually works.
- Last update timestamp (git commit)
- Hierarchical table of contents
- Brief summaries of each module/section (2-3 sentences max)
- Key topics and concepts per section
- Navigation structure

**Format**:
```
# Site Title
> Site description
> Last Updated: [git commit hash] [timestamp]

## Table of Contents
[Hierarchical listing]

## Module Name
Summary: [2-3 sentences]
Key Topics: [bullet points]

### Section Name
[Brief description - 1-2 sentences]
```

### llms-full.txt (Complete Version)
**Purpose**: Comprehensive reference with full content
**Max Size**: No limit, but should be well-structured
**Content**:
- Everything from llms.txt
- Full content from each MDX file
- All code examples with language tags
- Complete explanations and details
- Exercise descriptions
- Resource links

**Format**:
```
# Site Title
> Site description
> Last Updated: [git commit hash] [timestamp]

## Table of Contents
[Hierarchical listing with hyperlinks. Base URL is site URL from metadata]

---

## Module Name
[Full module overview content]

### Section Name
[Complete section content including all text, code blocks, examples]

---
```

## Content Processing Rules

1. **Preserve Structure**: Maintain the hierarchy from docs.json navigation
2. **Clean Markdown**: Remove MDX-specific syntax (imports, components)
3. **Code Blocks**: Preserve with language tags
4. **Links**: Convert relative to absolute where applicable
5. **Headings**: Normalize to standard markdown (#, ##, ###, etc.)
6. **Metadata**: Strip frontmatter but preserve key information
7. **Order**: Follow exact order from docs.json navigation

## Best Practices

1. **Consistency**: Use uniform formatting throughout
2. **Readability**: Optimize for both LLM parsing and human reading
3. **Completeness**: Ensure all public pages from docs.json are included
4. **Clarity**: Use clear section separators (---)
5. **Timestamps**: Include git commit info for version tracking
6. **No Duplication**: Avoid repeating content between sections
7. **Progressive Detail**: llms.txt provides overview, llms-full.txt provides depth

## Update Process

1. Check git for latest commit timestamp
2. Read docs.json for navigation structure
3. Process each MDX file in order
4. Generate llms.txt with summaries
5. Generate llms-full.txt with complete content
6. Validate both files are well-formed

When updating the llms files:
1. Always run `git log -1 --format="%H %ci"` to get latest commit info
2. Check docs.json for any navigation structure changes
3. Verify all new MDX files are included
4. Maintain consistent formatting with existing content
5. Update both llms.txt and llms-full.txt together
6. Test that files are valid markdown and well-formatted

## Content Guidelines

### For llms.txt (Concise)
- Keep module summaries to 2-3 sentences
- List only key topics (3-7 items max per section)
- Include production impact metrics when available
- Focus on "what" and "why", not detailed "how"
- Include quick reference tables and checklists

### For llms-full.txt (Complete)
- Include full explanations and context
- Preserve all code examples (both Python and TypeScript when available)
- Maintain production examples and real-world metrics
- Keep detailed implementation patterns
- Include complete workflows and decision trees

## Maintenance Checklist

When regenerating files:
- [ ] Git commit timestamp updated
- [ ] docs.json navigation structure followed
- [ ] All MDX files processed in order
- [ ] Code examples preserved with correct language tags
- [ ] Tables formatted correctly
- [ ] Links are valid (or converted to descriptions if relative)
- [ ] No MDX-specific syntax remaining (Card, CodeGroup, etc.)
- [ ] Section separators (---) used appropriately
- [ ] Both files validated as proper markdown
